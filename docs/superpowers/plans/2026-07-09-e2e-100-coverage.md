# E2E 100 Percent Coverage Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Raise the Playwright E2E coverage gate from the current 80% baseline toward 100% Monocart-reported source coverage while using uncovered code to find real product bugs and inconsistencies.

**Architecture:** Keep Monocart Coverage Reports as the only source of truth for browser E2E source coverage. Do not merge unit coverage, do not add custom percentage math, and do not create repo-specific coverage calculators. New work should either exercise reachable product behavior through E2E journeys, delete unreachable/dead code, or justify standard reporter exclusions for code that cannot execute in the Chromium E2E environment.

**Tech Stack:** Playwright, Chromium V8 coverage, Monocart Coverage Reports, npm scripts, GitHub Actions, TypeScript, React.

---

## Target Definition

The target is not "force every byte to execute by any means." The target is:

1. Monocart is the source of truth for source coverage.
2. `npm run test:e2e:coverage` remains the command that produces the CI badge and threshold result.
3. Reachable implemented browser behavior is exercised by Playwright E2E tests.
4. Unreachable code is handled by one of three actions:
   - remove it if it is dead code,
   - refactor it behind a reachable integration boundary,
   - exclude it only through standard Monocart/source-ignore mechanisms with a written reason.
5. No custom merge script or custom percentage calculation is introduced.

## Coverage Gate Strategy

Exact 100.00% byte coverage is a brittle CI gate while the app is still changing. Use staged gates:

- Current gate: 80%.
- Phase 1 gate: 85% after model setup, presenter, and storage journeys are stable.
- Phase 2 gate: 90% after animation, import/export, and prompting journeys are stable.
- Phase 3 gate: 95% after translation, mirror, AI image, and remaining service branches are stable.
- Final gate: 100% only after every remaining uncovered range is either exercised, deleted, or justified with a standard Monocart exclusion.

If the team wants a healthier long-term gate, keep global CI at 95% and separately require a "100% explained uncovered code" checklist. If the requirement is strict, proceed to 100% only after the exclusion review is complete.

- [ ] No custom coverage percentage math was added.
